AskUserQuestion TUI keystroke protocol:
```
Single choice (e.g. option 2 of 4):
  - Write "2" (number key)
  - Write "\n" (Enter to confirm)

Multi choice (e.g. toggle options 1 and 3, then submit, 4 total options):
  - Write "1" (toggle option 1)
  - Write "3" (toggle option 3)
  - Write "\x1b[B" repeated (option_count) times (Down arrow to reach Submit button)
  - Write "\n" (Enter to confirm)

Free text (Other option, 4 total options):
  - Write "\x1b[B" repeated (option_count) times (Down past all options to Other)
  - Write "\n" (Enter to select Other)
  - Write the text content
  - Write "\n" (Enter to confirm)
```

Answer payload from app:
```json
{

  "kuerzel": "session-name",
  "type": "single|multi|text",
  "selections": [1, 3],
  "text": "free text here",
  "option_count": 4
}
```

  <action>
Add a new `answer` action handler in the WebSocket `ws.on('message')` block of relay-server.cjs, alongside the existing `command` and `raw_command` handlers.

The handler receives: `{ action: "answer", kuerzel, type, selections, text, option_count }`

Create a function `computeKeystrokeSequence(msg)` that returns a string of characters to write:

For `type === "single"`:
- The keystroke is `String(msg.selections[0])` + `\n`
- Example: selections [2] -> "2\n"

For `type === "multi"`:
- First, write each selection number as a separate character: `msg.selections.map(n => String(n)).join('')`
- Then, write Down arrow (`\x1b[B`) repeated `msg.option_count` times to reach the Submit button
- Then write `\n` to confirm
- Example: selections [1,3], option_count 4 -> "13\x1b[B\x1b[B\x1b[B\x1b[B\n"

For `type === "text"`:
- Write Down arrow (`\x1b[B`) repeated `msg.option_count` times to reach "Other"
- Write `\n` to select Other
- Write the `msg.text` content
- Write `\n` to confirm
- Example: text "custom answer", option_count 4 -> "\x1b[B\x1b[B\x1b[B\x1b[B\ncustom answer\n"

The handler then calls `dispatchKeystrokesToPane(msg.kuerzel, keystrokes)` which is a new async function similar to `dispatchCommand` but does NOT append `\n` (the keystroke sequence already includes newlines where needed). It uses `findSessionForKuerzel` + `findPaneForTab` + `writeCharsToPane` like `dispatchCommand` does.

Add a 50ms delay between individual keystrokes for multi-select (between each number toggle) using a small helper that writes chars one-at-a-time with setTimeout. For single and text types, send the full string at once since no TUI state change happens between characters.

IMPORTANT: For multi-select, each number key must be sent as a separate `writeCharsToPane` call with a 50ms gap, because the TUI needs time to process each toggle. Build this as `async function dispatchMultiSelectAnswer(kuerzel, selections, optionCount)` that loops through selections with delays, then sends the Down arrows + Enter.

In the `wss.on('connection')` handler, after authentication succeeds and `appSocket` is set, add a call to a new function `sendReconnectSync()`.

`sendReconnectSync()` reads all `/tmp/zellij-claude-tab-*` files to build a list of active sessions (same logic as `findSessionForKuerzel` but collects all). For each file, the filename gives the Zellij session ID and the file content gives the kuerzel.

Then it sends a single `session_list` message to the app with all active sessions:
```javascript
appSocket.send(JSON.stringify({
  type: 'session_list',
  sessions: sessions.map(s => ({ name: s.kuerzel, active: true })),
  timestamp: Date.now()
}));
```

This reuses the same `session_list` message format that `listSessions()` already sends (see line ~193), so the app's existing session list parser handles it without changes.

The function should be called right after setting `appConnected = true` and `disconnectedAt = null` on line ~226, before the `ws.on('message')` handler. Wrap in try/catch so a failure doesn't break the connection.

Also call `sendReconnectSync()` with a short delay (500ms) via `setTimeout` to give the app time to register its message handlers after the WebSocket opens.
