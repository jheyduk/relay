    **NetworkMonitor.kt** (package: `dev.heyduk.relay.service`):
    Wraps ConnectivityManager.NetworkCallback into Flow&lt;Boolean&gt;.
    ```kotlin
    class NetworkMonitor(private val context: Context) {
        private val connectivityManager =
            context.getSystemService(Context.CONNECTIVITY_SERVICE) as ConnectivityManager

        val isConnected: Flow<Boolean> = callbackFlow {
            val callback = object : ConnectivityManager.NetworkCallback() {
                override fun onAvailable(network: Network) { trySend(true) }
                override fun onLost(network: Network) { trySend(false) }
            }
            val current = connectivityManager.activeNetwork != null
            trySend(current)
            connectivityManager.registerDefaultNetworkCallback(callback)
            awaitClose { connectivityManager.unregisterNetworkCallback(callback) }
        }.distinctUntilChanged()

        suspend fun awaitConnected() {
            isConnected.first { it }
        }
    }
    ```

    **DataStoreOffsetProvider.kt** (package: `dev.heyduk.relay.data`):

    class DataStoreOffsetProvider(private val dataStore: DataStore<Preferences>) : OffsetProvider {
        private val OFFSET_KEY = longPreferencesKey("polling_offset")

        override suspend fun getOffset(): Long =
            dataStore.data.first()[OFFSET_KEY] ?: 0L

        override suspend fun setOffset(offset: Long) {
            dataStore.edit { it[OFFSET_KEY] = offset }
        }
    }

    **PollingService.kt** (package: `dev.heyduk.relay.service`):
    Foreground service with type `dataSync`. Token flow is explicit -- tokens are read from DataStore
    at service start, NOT injected via Koin singletons.

    - Inject `NetworkMonitor`, `DataStore<Preferences>`, `OffsetProvider`, `RelayDatabase`, and `HttpClient` via Koin
    - Do NOT inject TelegramApi or TelegramPoller from Koin -- construct them locally with real tokens
    - `onStartCommand`:
      1. Create notification channel "relay_polling" with IMPORTANCE_LOW
      2. Build notification showing "Relay connected"
      3. Call `startForeground(NOTIFICATION_ID, notification, ServiceInfo.FOREGROUND_SERVICE_TYPE_DATA_SYNC)`
      4. Launch coroutine scope (SupervisorJob + Dispatchers.IO):
         a. Read tokens from DataStore synchronously: `val prefs = dataStore.data.first()`
         b. Extract `relayBotToken = prefs[stringPreferencesKey("relay_bot_token")]`
         c. Extract `commandBotToken = prefs[stringPreferencesKey("command_bot_token")]`
         d. Extract `chatId = prefs[stringPreferencesKey("chat_id")]`
         e. If any token is null or blank, log error via Timber, update notification to "Missing configuration", and `stopSelf()` -- do NOT start polling
         f. Construct `TelegramApi(httpClient, relayBotToken, chatId)` for relay bot
         g. Construct `TelegramPoller(relayApi, RelayMessageParser, offsetProvider)`
         h. Call `poller.pollLoop()` in a launched coroutine
    - Before each poll iteration, call `networkMonitor.awaitConnected()` -- wrap the pollLoop call in a loop that awaits connectivity first
    - Collect `networkMonitor.isConnected` in a separate coroutine to update notification text ("Connected" vs "Reconnecting...")
    - Implement `onTimeout(startId: Int, fgsType: Int)` for Android 15+: log warning, call `stopSelf()`
    - `onDestroy`: cancel coroutine scope
    - NOTIFICATION_ID = 1, CHANNEL_ID = "relay_polling"

    ```kotlin
    // Sketch of the token flow in onStartCommand:
    serviceScope.launch {
        val prefs = dataStore.data.first()
        val relayToken = prefs[stringPreferencesKey("relay_bot_token")] ?: ""
        val cmdToken = prefs[stringPreferencesKey("command_bot_token")] ?: ""
        val chatId = prefs[stringPreferencesKey("chat_id")] ?: ""

        if (relayToken.isBlank() || cmdToken.isBlank() || chatId.isBlank()) {
            Timber.e("PollingService: tokens not configured, stopping")
            stopSelf()
            return@launch
        }

        val relayApi = TelegramApi(httpClient = get(), botToken = relayToken, chatId = chatId)
        val poller = TelegramPoller(api = relayApi, parser = RelayMessageParser, offsetProvider = offsetProvider)

        // Collect updates and insert into DB
        launch {
            poller.updates.collect { update ->
                // Insert into SQLDelight messages table
                database.messagesQueries.insertOrIgnore(...)
            }
        }

        // Polling with network awareness
        while (isActive) {
            networkMonitor.awaitConnected()
            try {
                poller.pollLoop()  // Runs until error or cancellation
            } catch (e: CancellationException) { throw e }
            catch (e: Exception) {
                Timber.e(e, "Poll loop error, awaiting reconnect")
            }
        }
    }
    ```

    The SharedModule's TelegramApi/TelegramPoller singletons (which depend on these) will not resolve
    at app startup -- that is intentional. PollingService constructs its own TelegramApi and TelegramPoller
    instances with real tokens read from DataStore. The SharedModule's `TelegramRepository` singleton
    that depends on the poller will need to be wired differently -- the executor should update SharedModule
    to accept the poller as a late-bound dependency or have PollingService provide updates to the repository
    directly. The simplest approach: PollingService inserts updates into SQLDelight directly (shown in the
    sketch above), and TelegramRepository reads from SQLDelight only (no poller dependency for the DB path).
